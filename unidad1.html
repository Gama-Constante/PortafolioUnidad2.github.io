Unidad 1

Introducción al Desarrollo Web Orientado a Servicios

Fundamentos de SOA, arquitecturas de servicios web y conceptos esenciales de APIs

1. Arquitectura Orientada a Servicios (SOA)

La Arquitectura Orientada a Servicios (SOA) es un estilo de diseño de software donde las funcionalidades se ofrecen como servicios independientes y reutilizables que se comunican a través de una red mediante protocolos estándar.

¿Qué es SOA?

Definición

SOA es un paradigma de arquitectura para diseñar y desarrollar sistemas distribuidos donde la lógica de negocio se organiza en servicios autónomos que pueden ser descubiertos, accedidos y orquestados de manera flexible.

Características Principales de SOA

  • Servicios Débilmente Acoplados: Los servicios son independientes y pueden modificarse sin afectar a otros
  • Reusabilidad: Los servicios pueden ser utilizados por múltiples aplicaciones
  • Interoperabilidad: Funcionan en diferentes plataformas y tecnologías
  • Descubribilidad: Los servicios pueden ser encontrados y descritos mediante metadatos
  • Escalabilidad: Permiten escalar componentes individuales según necesidad
  • Autonomía: Cada servicio controla su propia lógica y datos

Componentes de SOA

Proveedor de Servicios

Crea y publica servicios disponibles

Registro de Servicios

Directorio donde se publican servicios

Consumidor de Servicios

Busca y utiliza los servicios

Bus de Servicios (ESB)

Facilita la comunicación entre servicios

Ventajas de SOA: Reutilización de código, mantenimiento simplificado, integración más fácil entre sistemas heterogéneos, mayor agilidad en el desarrollo y mejor escalabilidad del sistema.

2. Servicios Web

Los servicios web son componentes de software diseñados para soportar interacciones máquina-a-máquina sobre una red, utilizando estándares abiertos como HTTP, XML y JSON.

Tipos de Servicios Web

SOAP (Simple Object Access Protocol)

Protocolo basado en XML para intercambio de información estructurada en servicios web.

  • Usa XML para mensajes
  • Protocolo más formal y estricto
  • Soporta transacciones complejas
  • Ideal para operaciones críticas y empresariales
  • Incluye estándares de seguridad (WS-Security)
<?xml version="1.0"?> <soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope"> <soap:Body> <getUser> <userId>123</userId> </getUser> </soap:Body> </soap:Envelope>
REST (Representational State Transfer)

Estilo arquitectónico que utiliza HTTP y sus métodos para crear APIs web escalables.

  • Usa métodos HTTP (GET, POST, PUT, DELETE)
  • Formato flexible (JSON, XML, HTML)
  • Stateless (sin estado)
  • Más simple y ligero que SOAP
  • Ideal para aplicaciones web y móviles
  • Cacheable y escalable
// Ejemplo de solicitud REST GET /api/users/123 HTTP/1.1 Host: ejemplo.com Accept: application/json // Respuesta { "id": 123, "nombre": "Juan Pérez", "email": "juan@example.com" }

Comparación SOAP vs REST

Característica SOAP REST
Protocolo Protocolo estricto Estilo arquitectónico
Formato Solo XML JSON, XML, HTML, texto
Complejidad Más complejo Más simple
Rendimiento Más lento (XML pesado) Más rápido (JSON ligero)
Seguridad WS-Security integrado HTTPS, OAuth, JWT
Casos de uso Empresas, transacciones Web, móvil, APIs públicas
Recomendación: REST es preferido para la mayoría de aplicaciones web modernas por su simplicidad y eficiencia. SOAP se usa en entornos empresariales que requieren transacciones complejas y alta seguridad.

3. Principios de REST

REST (Representational State Transfer) se basa en un conjunto de principios arquitectónicos que permiten crear APIs web escalables, mantenibles y eficientes.

Los 6 Principios Fundamentales de REST

1. Cliente-Servidor

Separación entre la interfaz de usuario (cliente) y el almacenamiento de datos (servidor).

  • Permite desarrollo independiente del cliente y servidor
  • Mejora la portabilidad de la interfaz
  • Mejora la escalabilidad del servidor
2. Sin Estado (Stateless)

Cada petición del cliente al servidor debe contener toda la información necesaria.

  • El servidor no almacena contexto del cliente entre peticiones
  • Mejora la escalabilidad
  • Simplifica la implementación del servidor
3. Cacheable

Las respuestas deben indicar si pueden ser almacenadas en caché o no.

  • Reduce la carga del servidor
  • Mejora el rendimiento percibido
  • Usa headers HTTP (Cache-Control, ETag)
4. Interfaz Uniforme

Uso consistente de convenciones y estándares en toda la API.

  • Identificación de recursos mediante URIs
  • Manipulación de recursos mediante representaciones
  • Mensajes auto-descriptivos
  • HATEOAS (Hypermedia as the Engine of Application State)
5. Sistema en Capas

La arquitectura puede estar compuesta de capas jerárquicas.

  • El cliente no sabe si está conectado directamente al servidor final
  • Permite balanceadores de carga, proxies, gateways
  • Mejora la escalabilidad y seguridad
6. Código bajo Demanda (Opcional)

Los servidores pueden extender la funcionalidad del cliente enviando código ejecutable.

  • JavaScript, applets, etc.
  • Simplifica a los clientes
  • Principio opcional en REST

Métodos HTTP en REST

GET

Obtener recursos

POST

Crear nuevos recursos

PUT

Actualizar recursos completos

PATCH

Actualizar parcialmente

DELETE

Eliminar recursos

OPTIONS

Obtener opciones disponibles

// Ejemplos de endpoints RESTful GET /api/usuarios // Listar todos los usuarios GET /api/usuarios/5 // Obtener usuario con ID 5 POST /api/usuarios // Crear nuevo usuario PUT /api/usuarios/5 // Actualizar usuario 5 (completo) PATCH /api/usuarios/5 // Actualizar usuario 5 (parcial) DELETE /api/usuarios/5 // Eliminar usuario 5

4. Estándares de Servicios Web

Los estándares web son especificaciones técnicas que aseguran la interoperabilidad, seguridad y calidad de los servicios web.

Principales Estándares

WSDL (Web Services Description Language)

Lenguaje XML para describir servicios web y cómo acceder a ellos.

  • Define operaciones disponibles
  • Especifica parámetros de entrada y salida
  • Indica ubicación del servicio
  • Usado principalmente con SOAP
UDDI (Universal Description, Discovery and Integration)

Directorio para registro y descubrimiento de servicios web.

  • Registro centralizado de servicios
  • Clasificación y búsqueda de servicios
  • Metadatos de servicios
WS-Security

Estándar de seguridad para servicios web SOAP.

  • Autenticación de usuarios
  • Integridad de mensajes
  • Confidencialidad mediante cifrado
  • Firmas digitales
JSON Schema

Vocabulario para validar y documentar datos JSON.

  • Define estructura de datos JSON
  • Validación de tipos de datos
  • Documentación auto-generada
  • Usado en APIs REST modernas
OpenAPI (Swagger)

Especificación estándar para describir APIs RESTful.

  • Documentación interactiva de APIs
  • Generación automática de código cliente
  • Testing de endpoints
  • Estándar de la industria para REST
openapi: 3.0.0 info: title: API de Usuarios version: 1.0.0 paths: /usuarios: get: summary: Listar usuarios responses: '200': description: Lista exitosa content: application/json: schema: type: array items: $ref: '#/components/schemas/Usuario'

Formatos de Intercambio de Datos

JSON

Ligero, legible, estándar web

XML

Extensible, estructurado, validable

YAML

Legible, configuración

CSV

Tabular, simple, exportación

Mejores Prácticas: Usa OpenAPI/Swagger para documentar APIs REST, JSON como formato principal de datos, versionado de API (v1, v2), implementa autenticación (OAuth 2.0, JWT) y maneja errores consistentemente con códigos HTTP apropiados.
